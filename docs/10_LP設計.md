# LP設計

## サマリ

このドキュメントでは、「あと、」のランディングページ（LP）について、**何を意識して作ったか・各セクションの意図・デザイン方針・技術的な実装方針**をまとめる。LPの改修やA/Bテストを行う際の基準として使う。

## 1. LP全体の設計思想

### なぜLPを別サイトとして切り出したか

メインアプリ（`apps/frontend/web`）はログイン必須の認証フローを持つため、LPとは関心事が根本的に異なる。

| 観点 | メインアプリ | LP |
|---|---|---|
| 目的 | 書く体験の提供 | 興味を持った人を試用へ誘導 |
| 認証 | 必須 | 不要 |
| デプロイ先 | app.ato.so | ato.so |
| 更新頻度 | 開発に合わせて随時 | マーケ施策に合わせて随時 |

LPは独立してデプロイ・更新できることが重要であり、`apps/lp/` として分離した。

### 何が目的のページか

LPには1つだけのゴールがある。

> **「あと、を試してみようかな」と思ってもらい、サインアップページに到達させること。**

「プロダクトを理解させること」でも「全機能を説明すること」でもない。誤解を生まない最低限の情報で、最速でCTAに到達させる設計。

## 2. ターゲットとコンセプト

### 誰に向けたLPか

`09_マーケティング戦略.md` で定義したコアペルソナ「**テック感のある文系クリエイター**」に絞った。

具体的には：
- noteで定期的に発信している（エッセイ・ブログ・思考整理）
- PCとスマホ両方で書いているが、体験の分断にストレスを感じている
- NotionやObsidianを試したが「何かが違う」と感じて離脱した経験がある
- Gitは「聞いたことある」程度。使いたいが勉強はしたくない

このペルソナが「それ、自分のことだ」と感じたら、そのままCTAに流れる設計にした。

### 訴求の優先順位

マーケティング戦略から導き出した、ターゲットに最も刺さる訴求の順序：

1. **感情的共感**（「書くことが不自由だと感じたことはないか」）
2. **論理的差別化**（「あと、はこの3点が違う」）
3. **行動**（「無料で始める」）

## 3. セクション構成と設計意図

### 3-1. Nav（ナビゲーション）

```
[あと、]                              [始める]
```

**意図：**
- ロゴだけ。リンクは「始める」の1つだけ。
- 余分な情報を一切排除し、「書くこと」に集中するプロダクトの体験そのものをナビゲーションで体現する。
- スクロールに連れてグラスモーフィズム（`backdrop-filter: blur`）で背景をぼかし、コンテンツと視覚的に分離。

**判断の根拠：**
ナビゲーションにメニュー項目を増やすと、ユーザーの視線が分散しCTAクリック率が下がる。LPは1ページ完結なので、そもそもナビゲーションで移動する必要がない。

---

### 3-2. Hero（ファーストビュー）

```
noteで書いている人のための、下書き・思考整理エディタ。  ← 誰のためか

只々、自然に書くためのアプリ。  ← ブランドコピー（しっぽり明朝）

スマホでMarkdownがスムーズに書けて、
AI機能もGitHub連携もPCとの連携も全て同期された、
デバイスも場所も選ばないエディタ。  ← 何が違うか

[無料で始める]
データはいつでも手元に。  ← 安心感の補足
```

**意図：**
- **最初の一文で「誰のためか」を宣言する。** LPを開いた1秒で「自分には関係ない」か「自分のことだ」かが判断される。ターゲットではない人を早く離脱させ、ターゲットに集中する。
- **タグライン（しっぽり明朝）は感情を動かすために。** BIZ UDゴシックで書くとビジネス文書になる。しっぽり明朝で書くことで「美しい・好き・使ってみたい」という感情に訴える。
- **差別化メッセージはHeroに入れる。** スクロールしない人にも、他のツールと何が違うかを伝えるために。
- **「データはいつでも手元に。」はCTAの下に置く。** サインアップへの心理的障壁を下げるため。「あなたのデータは閉じ込められない」という安心感を、行動を促す直前に伝える。

**コピーの変遷（なぜこの文章になったか）：**

当初の「何が違うのか」の候補：

| 案 | 問題点 |
|---|---|
| 「Markdownで書いて、ZIPで出せる。スマホでもPCでも、同じ体験で。」 | 機能を羅列しているが、AI・GitHub連携がLP公開時点でリリース済みのため訴求に含めたい |
| 「スマホでMarkdownがスムーズに書けて〜デバイスも場所も選ばないエディタ。」 | LP公開後に機能が揃っている前提で、強みを網羅的に伝えられる |

→ LP公開のタイミングはAI機能・GitHub連携の実装後を前提としているため、後者を採用。

### 3-3. 4つの壁（問題提起）

```
書くことが、なぜこんなに不自由なのか。

[場所の壁]        [疲れるという壁]
[サービスの壁]    [機能の壁]
```

**意図：**
- **ソリューション提示の前に、問題の共有が必要。** 「あと、が解決します」とだけ言っても、問題を認識していない人には響かない。「そうそう、これが嫌だったんだ」という共感を先に作る。
- **グレー背景（白金）で視覚的に区切る。** 問題提起は感情的に重い内容なので、Heroの白い空間と区別して「モード転換」を演出する。
- **カード形式で4つ並べる。** リスト形式より読みやすく、スマホでも2列→1列にレスポンシブする。

---

### 3-4. あと、が違うこと（解決策）

```
あと、が違うこと。
4つの壁を、ひとつのエディタで。

01  データの自由
    Markdownで書いて、ZIPで持ち出せる。...

02  どこでも、途切れない
    電車でスマホに書いたものが、...

03  機能は強力でも、勉強いらない
    Gitを知らなくてもバージョン管理ができる。...
```

**意図：**
- **3つに絞った。** 強みをすべて伝えようとしない。「データの自由・シームレスな同期・学習コストゼロ」の3点が、4つの壁に対応する最も刺さるメッセージ。
- **番号（01/02/03）をつけた。** 読者に「3つあるんだ」と事前に認知させ、最後まで読むモチベーションを作る。
- **「データの自由」を01に置いた。** マーケティング戦略で「キラーフィーチャー」と定義したため。Notionのロックインに不満を持つ層への直球訴求。
- **白背景で前後と区別。** 問題提起（グレー）→ 解決策（白）→ ポジショニング（グレー）と交互にすることで、視覚的にリズムを作る。

### 3-5. noteとの関係（ポジショニング）

```
noteで書く人のための、相棒として。

[下書き・思考整理]      →      [清書・公開]
     あと、                        note
```

**意図：**
- **「あと、はnoteの競合ではなく補完ツールだ」と示す。** noteユーザーは「noteをやめてこっちに来い」と言われると警戒する。「今のワークフローに差し込むだけ」という安心感が必要。
- **図式化することで直感的に伝える。** 説明文だけだと抽象的になる。視覚的なフロー図にすることで、使い方のイメージを1秒で伝える。

---

### 3-6. コンセプト（哲学・ブランドの深み）

```
CONCEPT

心が動いた時。
アイデアを閃いた時。
その場で、そのまま書く。

道教の「無為自然」を指針に。
紙とペンのように、あるがままに書ける道具。
それが「あと、」のコンセプト。
```

**意図：**
- **感情的な購買判断を最終確認させる場所。** 機能や差別化の説明を終えた後、「なぜこのプロダクトが存在するのか」という物語を伝える。
- **noteユーザーは「作者の人間性を見てから使う」文化がある。** 機能の羅列だけでは信頼されない。哲学を持った作者が作るツールだと感じてもらうことで、最後のCTAへの心理的距離を縮める。
- **しっぽり明朝を使う。** 情緒的な内容であることを、フォントで表現する。

---

### 3-7. Final CTA

```
只々、自然に書くためのアプリ。

[無料で始める]
データはいつでも手元に。
```

**意図：**
- **タグラインを再掲してLPを締める。** 最初のHeroとの首尾一貫性を作り、「このプロダクトは何か」をもう一度確認させる。
- **CTAは2箇所（Hero・Final）のみ。** 途中のセクションにCTAを散りばめると読み物としての流れが壊れる。「話を最後まで聞いて、使ってみたくなった人」だけが押せばいい。

---

### 3-8. Footer

```
あと、                    © 2026 ato. All rights reserved.
```

**意図：** 最小限。信頼感のための著作権表示とブランド名のみ。

## 4. デザイン方針

### デザインシステムへの準拠

`00_プロダクトデザイン.md` のデザインシステムに完全準拠。

| 要素 | 適用内容 |
|---|---|
| フォント | BIZ UDゴシック（本文）/ しっぽり明朝（キャッチコピー・ブランド名・情緒的な文言） |
| カラー | 白(#fffffe)・墨(#373737)・白金(#e4e5e6)・薄墨(#bdbebe)のみ使用。アクセントカラーは未使用（CTAも墨） |
| スペーシング | 8px基準。セクション間は96px、コンポーネント間は24/32/48px |
| 角丸 | `border-radius: 8px` 統一 |
| レスポンシブ | `auto-fit, minmax()` でグリッドが自然にリフローする設計 |

### カラーについての判断

CTA（「無料で始める」）を青（露草）ではなく**墨（#373737）**にした。

理由：
- 「自然に書くためのアプリ」というコンセプトに、目立つ色でアピールする行為が矛盾する
- 全体が白と墨で統一されているため、ボタンも同系統にした方がブランドの統一感が出る
- 周囲が白背景の場合、墨ボタンは十分なコントラストを持つ

### タイポグラフィについての判断

| 使用箇所 | フォント | 理由 |
|---|---|---|
| ブランド名「あと、」 | しっぽり明朝 | ブランドアイデンティティ。道教・和の雰囲気を体現 |
| キャッチコピー | しっぽり明朝 | 感情に訴える。機能説明ではなく「詩」として読ませる |
| 本文・UI要素 | BIZ UDゴシック | ユニバーサルデザイン。見やすさ・読みやすさ優先 |
| 番号（01/02/03） | BIZ UDゴシック | 機能的な情報なので読みやすさ優先 |

## 5. 技術的な実装方針

### スタック

| 項目 | 採用 | 理由 |
|---|---|---|
| フレームワーク | Next.js 15 | メインアプリと統一。SSG/SSRの選択肢を持てる |
| スタイリング | インラインスタイル | 既存コードベースとの一貫性。CSSビルドの依存を増やさない |
| フォント | next/font/google | フォントの最適化（preload・display swap）を自動処理 |
| 状態管理 | なし | LPはすべて静的コンテンツ |

### デプロイ想定

```
ato.so        → apps/lp/ （このLP）
app.ato.so    → apps/frontend/web/ （メインアプリ）
```

CTAのリンク先は `https://app.ato.so/login` とし、ドメインをまたいで誘導する。

### 今後の拡張

| 優先度 | 内容 |
|---|---|
| 高 | スマホ表示の細かい調整（Heroの文字サイズ・改行位置） |
| 高 | OGP画像の追加（SNSシェア時の見た目） |
| 中 | `prefers-color-scheme` によるダークモード対応 |
| 中 | アニメーション追加（フェードイン・スクロールトリガー） |
| 低 | A/Bテスト（Heroコピー・CTAテキストの比較） |

## 6. やらなかったこと・判断の記録

| 検討したこと | やらなかった理由 |
|---|---|
| 料金プランセクション | LP公開時点では無料のみ。有料プランが決まってから追加する |
| スクリーンショット・デモ動画 | エディタのUIが未完成のため、見せると逆効果になる可能性がある |
| 競合比較表 | noteユーザーはツールの比較より「作者の思想」で判断する層。比較表はビジネス色が強くなる |
| FAQ | 現時点では「試してみれば分かる」の姿勢を優先。問い合わせが増えてから対応 |
| SNSリンク | 発信場所が散らばることで、ユーザーのアクションが分散する。CTAへの集中を優先 |
