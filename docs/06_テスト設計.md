# テスト設計

## サマリ

このドキュメントでは、テストレベル（ユニット/結合/E2E）、対象範囲、主要観点（境界/異常系）、テスト項目一覧、データ準備/環境、合格基準を定める。CRDTによる自動マージ同期を前提とし、競合テストは自動マージ結果の検証に集約。

## 本文

### テストレベル

#### テストピラミッド

```
      /   \
     / E2E \       少量・高コスト・高信頼性
    /───────\
   /  Integ  \     中量・中コスト
  /───────────\
 /     Unit    \   大量・低コスト・高速
/───────────────\
```

#### 各レベルの定義

| レベル | 対象 | ツール | 実行タイミング |
| --- | --- | --- | --- |
| E2E | ユーザーシナリオ全体 | Playwright | CI(PR時) |
| 結合 | モジュール間連携・API | Vitest, Testcontainers, Hono `app.request()` | CI(PR時) |
| ユニット | 関数・コンポーネント単体 | Vitest, React Testing Library | 開発環境・CI(PR時) |

#### テスト比率の目標

| レベル | 比率 | 理由 |
| --- | --- | --- |
| E2E | 10% | クリティカルパスの保証 |
| 結合 | 20% | API契約・DB連携の検証 |
| ユニット | 70% | 高速フィードバック・リファクタリング安全性 |

#### 対象範囲

##### フロントエンド

| カテゴリ | 対象 | テストレベル |
| --- | --- | --- |
| ドメインロジック | バリデーション, ツリー操作 | ユニット |
| CRDT操作 | FileTreeDoc, FileContentDoc, Yjs操作 | ユニット + 結合 |
| Hooks | useEditor, useFileTree, useSync, useAuth | ユニット |
| コンポーネント | Editor, FileTree, ProjectSelector, Modal | ユニット |

| API連携 | REST Client（SyncManager, AuthManager） | 結合 |
| 同期状態 | SyncState（5状態遷移） | ユニット |
| 画面遷移 | 認証フロー, プロジェクト操作 | E2E |

##### バックエンド（API）

| カテゴリ | 対象 | テストレベル |
| --- | --- | --- |
| ハンドラー | AuthHandler, ProjectHandler, SyncHandler | ユニット + 結合 |
| ミドルウェア | 認証, RLS設定, エラーハンドリング | ユニット |
| リポジトリ | DB操作（CRDT保存/取得） | 結合 |
| 同期処理 | CRDT更新の受信・保存・配信 | 結合 |

##### データベース

| カテゴリ | 対象 | テストレベル |
| --- | --- | --- |
| 制約 | UNIQUE制約, 外部キー | 結合 |
| RLS | ポリシー動作, ユーザー分離 | 結合 |
| CRDTスナップショット | file_tree_docs/file_content_docs保存/取得 | 結合 |

#### 除外範囲

| 対象 | 理由 |
| --- | --- |
| サードパーティライブラリ内部（Yjs等） | ライブラリ側でテスト済み |
| CSSスタイリング詳細 | ビジュアルリグレッションは初期リリース後に検討 |

#### 初期リリース後に実施予定

以下は初期リリースでは簡易的な確認にとどめ, リリース後に本格実施する。

| 対象 | 初期リリース時 | リリース後 |
| --- | --- | --- |
| 性能テスト | 主要操作の手動計測（レスポンス100ms以内, 初回ロード3秒以内） | 自動化された性能テストスイート |
| ブラウザ互換性 | 主要ブラウザ最新版+1つ前で手動確認 | BrowserStackによる網羅的検証 |
| セキュリティスキャン | npm audit, 依存関係の脆弱性チェック | 外部セキュリティ監査 |

### 主要観点（境界/異常系）

#### 境界値テスト

##### 入力値の境界

| 対象 | 境界値 | テスト内容 |
| --- | --- | --- |
| 名前（プロジェクト/ファイル/フォルダ共通） | 1文字, 255文字, 256文字 | 最小・最大・超過 |
| ファイルコンテンツ | 0バイト, 大容量 | 空・大きなデータ |

※バリデーションルールは`要件定義.md`を参照

##### システム制限の境界

| 対象 | 境界値 | テスト内容 |
| --- | --- | --- |
| ローカルストレージ | 5MB付近 | 容量警告・超過 |
| リトライ回数 | 3回, 4回目 | 最大リトライ・超過でダイアログ表示 |

#### 異常系テスト

##### ネットワーク異常

| シナリオ | 期待動作 |
| --- | --- |
| API呼び出し中にオフライン | オフライン状態に遷移, CRDT更新をローカル保持 |
| タイムアウト | リトライ（指数バックオフ） |
| 5xxエラー | リトライ（最大3回）, retryable: true |
| 4xxエラー | リトライなし, retryable: false, エラー通知 |
| 3回リトライ失敗 | DialogOpen状態, ユーザーに選択肢提示 |

##### 認証異常

| シナリオ | 期待動作 |
| --- | --- |
| アクセストークン期限切れ | リフレッシュトークンで自動更新 |
| リフレッシュトークン期限切れ | ログイン画面へリダイレクト, ローカルデータ削除 |
| 無効なトークン | ログイン画面へリダイレクト |
| OAuth認証キャンセル | ログイン画面に戻る |
| オフライン中のトークン失効 | オンライン復帰後に401で強制ログアウト |

##### ストレージ異常

| シナリオ | 期待動作 |
| --- | --- |
| localStorage容量超過（QuotaExceededError） | 容量超過ダイアログ表示（✕なし, 操作必須）, 「ファイルを削除」or「エクスポート」選択 |
| 破損データ検出 | サーバーから再取得 |

##### データ整合性異常

| シナリオ | 期待動作 |
| --- | --- |
| 存在しないprojectIdでの同期 | PROJECT_NOT_FOUND (404) |
| 他ユーザーのprojectIdでの同期 | RLSで拒否 |
| targetIdがprojectIdに属さない | VALIDATION_INVALID_TARGET (400) |

##### ファクトリパターン

テストの再現性と並列実行の安定性のため、決定的なID生成で大量のテストデータを作成可能にする。

```typescript
// シーケンス管理（テストごとにリセット）
let sequence = 0;
const resetSequence = () => { sequence = 0; };
const nextSequence = () => ++sequence;

// 決定的なUUID v4生成（テスト用）
const testUUID = (category: number) => {
  const seq = nextSequence();
  const hex = (n: number, len: number) => n.toString(16).padStart(len, '0');
  return `${hex(category, 8)}-${hex(seq & 0xffff, 4)}-4000-8000-${hex(seq & 0xffffffffffff, 12)}`;
};

// 固定タイムスタンプ（テスト用）
const TEST_TIMESTAMP = new Date('2024-01-15T00:00:00.000Z');

// カテゴリ定数
const UUID_CATEGORY = {
  USER: 1,
  PROJECT: 2,
  FILE: 3,
} as const;

// ユーザーファクトリ
const userFactory = {
  build: (overrides?: Partial<User>): User => {
    const seq = nextSequence();
    return {
      id: testUUID(UUID_CATEGORY.USER),
      googleId: `google-${seq}`,
      email: `test-user-${seq}@example.com`,
      name: `Test User ${seq}`,
      createdAt: TEST_TIMESTAMP,
      updatedAt: TEST_TIMESTAMP,
      ...overrides,
    };
  },
};

// プロジェクトファクトリ
const projectFactory = {
  build: (ownerId: string, overrides?: Partial<Project>): Project => {
    const seq = nextSequence();
    return {
      id: testUUID(UUID_CATEGORY.PROJECT),
      ownerId,
      name: `Test Project ${seq}`,
      createdAt: TEST_TIMESTAMP,
      updatedAt: TEST_TIMESTAMP,
      ...overrides,
    };
  },
};
```
