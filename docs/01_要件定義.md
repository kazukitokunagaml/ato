# 要件定義

## サマリ

このドキュメントでは、機能要件（ユーザーストーリー、ユースケース、操作ルール、設定項目）、非機能要件（性能、可用性、同期・異常系、対応環境、ストレージ、セキュリティ）、システム構成方針を定める。ローカルファーストでCRDT（Yjs）による自動マージ同期を採用し、Next.js/Vercel、Cloudflare Workers/Hono、Neon PostgreSQLで構成する。

## 本文

### 機能要件

#### ユーザーストーリー(ユーザー視点でのやりたいこと)

##### 場所の壁

- デバイスを換えても作業状況を維持したい
- 直近の操作履歴（undo/redo）をデバイス間で共有したい（将来対応）
- あらゆるデバイスとブラウザで使いたい
- 各デバイスの画面サイズに最適化された状態で使いたい
- オフラインでも作業したい

##### 疲れるという壁

- ガイドを読みたくないし、使い方で迷いたくない
- メチャクチャに触ってもバグらないでほしい（安心して使いたい）
- 作業中に不要な情報は出さないでほしい（作業に集中したい）
- フィードバックの遅延を起こさないでほしい（作業に集中したい）

##### サービスの壁

- いつでも(オフラインでも)インポート/エクスポートできてほしい
- 外部ツールと連携しやすくしてほしい

##### 機能の壁

- Gitを勉強しなくてもバージョン管理がしたい（将来対応）
- 生成AIを勉強しなくてもAIエージェントで執筆したい（将来対応）

##### その他

- 入力内容を暗号化してほしい（将来対応）
- 入力内容をAI学習させないでほしい（将来対応）
- 情報消失やサービス停止がないようにしてほしい（将来対応）

#### スコープ

- `初期リリース`
  - 画像のアップロード・管理機能はなし（URLによる埋め込みのみ対応）
  - バージョン管理機能はなし
  - AI連携機能はなし

##### プロジェクト管理

- `UCPJ-01: プロジェクト読み込み`
  - **アクター**
    - ユーザー
  - **前提条件**
    - ログイン済み
    - ※プロジェクト未選択でも実行可能
  - **シナリオ**
    1. リロード
    2. キャッシュを更新する
      1. (キャッシュがない場合)
        1. (プロジェクト一覧が1件以上)1番目のプロジェクトが対象になる
        2. (プロジェクト一覧が0件)プロジェクト作成が必要な旨を表示する
    3. キャッシュからUIが表示される
  - **代替例外**
    - オフラインでキャッシュがない場合は、通信復帰が必要な旨を表示する
  - **事後条件**
    - 対象プロジェクトが選択状態になる

- `UCPJ-02: プロジェクト切り替え`
  - **アクター**
    - ユーザー
  - **前提条件**
    - ログイン済み
  - **シナリオ**
    1. プロジェクト一覧から選択する
    2. 旧プロジェクトのローカルデータを削除する
    3. `プロジェクト読み込み`が自動実行される
  - **代替例外**
    - なし
  - **事後条件**
    - 新プロジェクトが選択状態になる

- `UCPJ-03: プロジェクト新規作成`
  - **アクター**
    - ユーザー
  - **前提条件**

  - **シナリオ**
    1. プロジェクト一覧から作成モードを開く
    2. 名前を入力して作成実行ボタンを押す
    3. `プロジェクト切り替え`が自動実行される
  - **代替例外**
    - なし
  - **事後条件**
    - 新プロジェクトが選択状態になる

- `UCPJ-04: プロジェクト名称更新`

    2. プロジェクト一覧から更新モードを開く

### 非機能要件

#### 性能

- `ファイル編集のレスポンス`
  - 入力から表示まで100ms以内（体感的に即時）

- `初回ロード時間`
  - 3秒以内（キャッシュあり：1秒以内）

- `同期処理`
  - バックグラウンドで非同期実行（UI操作をブロックしない）

#### 可用性

- `オフライン動作`
  - ネットワーク切断時も編集・閲覧が可能

- `PWA対応`
  - ブラウザ再起動後もオフラインで起動可能

- `データ永続性`
  - ローカルファースト方式によりブラウザを閉じても編集内容を保持

- `同期リトライ`
  - 同期失敗時は最大3回自動リトライ

#### 同期・異常系

- `同期動作`
  - ローカルとサーバーのデータ差分を双方向に同期
  - 同期後は必ずローカルとサーバーが同一状態になる

- `オフライン時`
  - ローカルデータに操作が蓄積され、オンライン復帰時に自動同期

- `同期失敗時の復旧`
  - ローカルデータに未同期の操作を保持し、次回オンライン時に自動リトライ
  - リトライ上限超過時はエラーダイアログを表示

※同期状態モデルの詳細は`アーキテクチャ設計.md`を参照。

#### 対応環境・互換性

- `対応ブラウザ`
  - Chrome: 最新版および1つ前のメジャーバージョン
  - Safari: 最新版および1つ前のメジャーバージョン
  - Firefox: 最新版および1つ前のメジャーバージョン
  - Edge: 最新版および1つ前のメジャーバージョン

- `対応OS`
  - Windows、macOS、iOS、Android（各最新版）

- `PWAインストール条件`
  - HTTPS接続かつService Worker対応ブラウザ

#### ローカルストレージ

- `容量制限`
  - ブラウザの上限（5-10MB）に依存
  - 将来的にIndexedDBに移行予定

- `容量超過時の挙動`
  - 通知: ユーザーに容量不足を通知し、エクスポートと削除を推奨
  - 制限: 新規作成/編集は不可、閲覧・削除・エクスポートは可能

#### セキュリティ

- `認証方式`: 外部認証基盤に責任を委譲、開発環境のみワンクリックで認証
- `データ分離`: RLSによりユーザー間でデータを完全分離
- `セッション管理`: 認証トークンの自動リフレッシュ、失効時はログイン画面へリダイレクト
- `ローカルストレージ`: 暗号化なし（鍵を安全に置ける場所について目処がついていない）
- `XSS対策`: ユーザー入力による不正なスクリプト実行を防止（自動エスケープなど）
- `監査ログ`: 初期リリースではCloudflare Workers Logsのみ

※詳細は`セキュリティ設計.md`を参照。

### システム構成

#### 方針

円滑なスケールや改修コスト抑制のため、初期段階から大規模化を見越した構成で開発する。

#### 構成

- `フロントエンド`
  - 構成
    - Next.js
    - Vercelホスティング
  - 採用理由
    - 実機テストを含むプレビュー環境の提供が容易
    - Next機能もフルで活用できる

- `バックエンド(サーバーレスAPI)`
  - 構成
    - Cloudflare Workers
    - Hono(Workers互換フレームワーク)
  - 採用理由
    - 同期・競合解決の挙動をAPI側で厳密に制御できる
    - 低運用でスケール可能
    - Honoはモノレポ時にAPIの型をフロントに共有可能

- `DB`
  - 構成
    - Neon(PostgreSQL)
  - 採用理由
    - 標準技術に寄せて、将来的なロックインを回避可能
    - バックアップやバージョン管理が容易

- `CICD`
  - 構成
    - GitHub Actions
  - 採用理由
    - GitHubとの統合がシームレスで設定が容易
    - Vercel/Cloudflareとの連携が標準でサポートされている

- (現時点では不要)`ジョブ/キュー`
  - 構成
    - Cloudflare Queues
  - 採用理由
    - AIなどの重い処理をクライアントが閉じられた後も継続可能

- (現時点では不要)`リアルタイム`
  - 構成
    - Cloudflare Durable Objects（WebSocket）
  - 採用理由
    - 低遅延の編集体験を実現
